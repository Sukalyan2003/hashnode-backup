## Starting with Go

At this step, I first go through some tutorials to orient myself, and i’ll note down some interesting things i’ve noticed in this section. This section is not a tutorial of Go, i am learning these things myself for the first time. Refer to the links in the references to learn.

`go mod tidy` is used to manage dependencies. Noting it here cause tidy is a bit less obvious to remember than init and other stuff.

The declaration order is variable name and then datatype and then return type (if it’s a function). This again is different than the languages I’m used to.

important: If your variable or function name is small cased, then it is not an “Exported Name” which is another way of saying that it’s a private type. To make it accessible outside the immediate scope, always start with a Capital Letter.

The testing here is a little bit of new syntax but works essentially the same as testing in Java or python.

The package of the test must be same as the package of the code it is testing.  
In Go, any function in a `*_test.go` file whose name starts with `Test` is treated as a test by `go test`.

One particular concept that i found really interesting was the concept of `defer` in Go. At first I thought it was something like `yield` in Python, but on closer inspection it's a totally different thing.  
We use yield to stop the execution of the function, do some side effects and then come back to where the function stopped; While the defer command is usually used as a Stack of cleanup operations.

Instead of manually keeping track of closing files/mutexes, other important things, we defer their cleanup right below where they are first used, and when the function around these is done executing, all these defer statements get executed.

## Defining our protocol

Even before we can write the first line of code, we should figure out exactly what we want it to do. This will be the recurring theme here, because instead of building something quickly and moving on, we want to learn the internals.

Every server needs a Transport protocol, that will decide how the data itself will be sent at the lower levels, in our case, that will be the TCP Model, at the application level.

There are many different ways to implement this, especially in the part about Message Framing, but in the end, I chose a Text based and line based system first, to make the debugging easier, with future plans to shift it to binary if needed.

While reading up on this, I learned something totally unexpected, that a lot of Binary Data is actually encoded to Base64 to make storage and transmission easy in Text based systems, without requiring major rewrites to support raw binary data as in BLOB files.

### So what is a Protocol?

A protocol is essentially the what, when and how of transporting data over a network. There are three main components of it:

1.  The syntax of the communication: how do we phrase the information we exchange.
    
2.  The semantics of the communication: what is the proper response for each information piece received.
    
3.  The synchronization of the communication: whose turn it is to speak (given the above defined semantics).
