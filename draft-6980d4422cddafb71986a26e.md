### Phase 2 - Making it more reliable

Establish clear connection parameters such as Polling rate, timeouts and what to do if a connection crashes.

pprof used for performance profiling along with proper error handling code.

A simple script that sends a mix of GET and PUT requests at rates of 10/100/1000 concurrently.

Adding Request IDs for each request sent. This is to maintain idempotency. Also explain in the post why idempotency is important for databases.

By maintaining an in-memory LRU Cache of all request IDs we can ensure that no request gets duplicated.

Fine tuning the connection parameters based on the results of the load testing done at the end of the previous phase.

Take some steps to reduce Backpressure. (Think about Isolation and locking strategies)

Repeat the load test with profiler.

### Phase 3 - Introducing Sharding

Explain in detail the context, theory and real world uses of sharding, then add that to your database.

Let there be multiple servers running together now, with each of them having their own SQLite database (each one is one shard)

Implement a router service that quickly figures out which shard has the data that the client is requesting for. This can also be a bottleneck but client side shard calculation does not seem good to me. Figure out how to make it as fast as possible.

To ensure consistency over availability, whenever a shard fails, crash the server.

Here, the only goal is to reliably make sure that the Shards are backed up with atleast one replica for each but configurable for as many as we might need.
