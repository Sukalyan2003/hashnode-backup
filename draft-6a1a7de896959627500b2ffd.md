## A little Review

A basic RAG system follows a fairly straightforward pattern: take the user’s query, retrieve relevant chunks from a knowledge base, pass those chunks to an LLM, and generate an answer. This approach is powerful because it gives the model access to external and domain-specific knowledge instead of relying only on what it learned during training.

But it also has an important limitation.

Traditional RAG is mostly reactive. It responds to the user’s question in a single pass. It does not truly “think” about whether the query needs to be rewritten, whether the retrieved documents are sufficient, whether another source should be consulted, whether the answer should be verified, or whether a tool should be called before responding. In many cases, this is enough. For simple factual questions, basic RAG can work very well.

However, real-world problems are rarely that clean.

Users ask vague questions. They mix multiple intents in one prompt. They refer to earlier conversations. They expect answers that depend on structured data, unstructured documents, APIs, tools, personal memory, and sometimes even live external systems. The information needed to answer a question may not be present in one clean document chunk. It may be spread across multiple files, hidden behind relationships between entities, or dependent on business rules that require reasoning.

This is where Agentic RAG becomes important.

## Agentic RAG

Once we add agency to RAG, the system begins to look less like a simple pipeline and more like a coordinated workflow. There may be a planner agent that breaks the user’s request into smaller tasks. There may be a retrieval agent that decides how to search across vector databases, keyword indexes, SQL databases, APIs, or graph stores. There may be a reasoning agent that combines evidence from multiple sources. There may be a critic agent that checks whether the answer is grounded in the retrieved context. There may even be a memory component that stores useful long-term information about the user, the domain, or previous interactions.

Some systems use a single agent that manages retrieval, tool use, and response generation. Others use multiple specialized agents that collaborate. Some rely heavily on reflection, where the system evaluates and improves its own output. Some introduce planning, where the agent decides the sequence of actions before answering. Some use tool-augmented workflows, where the LLM can call search tools, databases, calculators, APIs, or code execution environments. And some extend RAG further by combining agents with knowledge graphs, creating what is often described as Agentic GraphRAG.

A production-grade Agentic RAG platform needs orchestration, memory, observability, error handling, fallback strategies, evaluation, and guardrails. It needs to know not only how to retrieve information, but also how to recover when retrieval fails. It needs to trace what happened during a conversation so developers can debug the agent’s behavior. It needs to separate short-term conversation context from long-term memory. It needs to decide when to trust a retrieved document, when to search again, and when to admit that it does not have enough information.

Similarly, vector databases such as Qdrant provide the retrieval layer needed for semantic search, while tool integration patterns such as MCP-style interfaces help agents interact with external systems in a more standardized way. Memory extraction allows the system to preserve useful information beyond a single conversation. Observability tools help track failures, latency, retrieval quality, and decision paths. Together, these pieces show that Agentic RAG is not just about making the LLM “smarter.” It is about designing a reliable AI system around the LLM.

### Graphs

Traditional vector-based RAG works by searching for semantically similar chunks of text. This is useful, but it often treats knowledge as isolated pieces of information. In many domains, the relationships between facts are just as important as the facts themselves. For example, in a financial investigation, it may not be enough to retrieve documents about a company. We may need to understand how that company is connected to subsidiaries, executives, transactions, regulations, and historical events. In a healthcare setting, the answer may depend on relationships between symptoms, medications, diagnoses, contraindications, and patient history. In an enterprise system, the answer may depend on how teams, projects, services, incidents, and documents are connected.

A knowledge graph represents these relationships directly.

When we combine graphs with agents, the system can reason over connected information rather than only retrieve similar text. An agent can follow relationships, perform multi-hop reasoning, identify relevant entities, traverse the graph, and combine graph-based evidence with document-based evidence. This gives the system a stronger foundation for questions that require context, relationships, and structured reasoning.

This is where frameworks such as LangGraph become useful. Instead of treating the application as a linear chain, a graph-based orchestration framework allows the developer to model the agent workflow as a set of nodes and transitions. One node may rewrite the query. Another may perform retrieval. Another may call a tool. Another may validate the response. Another may decide whether the system should continue reasoning or stop and answer the user. This makes the agentic workflow more explicit, more controllable, and easier to debug.

### Tradeoffs

Agentic systems are more powerful, but they are also more complex. They can be slower because they may perform multiple reasoning and retrieval steps. They can be harder to evaluate because the path to an answer is dynamic. They can be more expensive because each tool call or reasoning step may involve additional model usage. They can also be more difficult to control because the system is making decisions at runtime rather than following a fixed pipeline.

So the real question is not simply: “Should we use Agentic RAG?”

The better question is: “Which parts of the RAG pipeline actually need agency?”

## Project Updates

In the previous project update, I barely gave you an overview of the several different components.

If I had to summarize the entire state of the project right now, it would be as this:

```plaintext
A local-first Retrieval-Augmented Generation chatbot. 

Documents (PDF, DOCX, TXT, MD, JSON, CSV, web URLs) are chunked, embedded against an Ollama model, and stored in a pickled cache.

At query time the user's question is validated and analyzed, the top-K most relevant chunks are retrieved by a hybrid of dense (cosine) and sparse (BM25) search fused with reciprocal rank fusion.

The LLM (also Ollama by default) is prompted with the context plus role-conditioned system prompt, and the answer is returned with source attribution. 

The whole engine is also exposed as a Model Context Protocol server over JSON-RPC stdio so any MCP-compatible client can drive it, and there are two Streamlit UIs — one talking to the MCP server, one importing the engine directly.
```

The core logic, is in the `src/core/final_rag.system.py`, which acts as an orchestrator. When you open it, the first thing you will see after the imports and some callback helper function, is the class of `FinalRagChatbot`.

Now this was a name i came up with out of frustration as I kept procrastinating my previous projects, but there are still many many phases left to be improved and further worked upon.

The class itself calls the functions for all the default parameter tuning, and cache path resolution and more. Contains all the methods needed to make the main chat loop work.

The main section of the code is the entire chat turn, and if we understand that, we can branch out into the several components part by part to understand them specifically.

There are some tracking and metrics related thing at the top of this function but the main logic starts as follows:

1.  Validating the input
    
    against harmful patterns like code execution; or when someone asks a security related question but does not have the authorization, via the `validate_input` and `check_permissions` functions.  
    Both of these are keyword based, might try some further semantic checks here later.
    
2.  Then we update the statistics  
    by incrementing the query number and sending a message that shows that it's processing. From here we start changing the state/stage of the processing using the callbacks declared above.
    
3.  Then the `analyze_query` function  
    uses some hardcoded rules and heuristics to convert potentially vague queries, and expand them out into better representations.
    
    For example, expanding out certain popular acronyms, check against common keywords to figure out whether it's asking for a tutorial or a report or troubleshooting etc, Isolate out the entities like Proper nouns and acronyms.
    
    And based on all this, it adds some extra lines of instruction to make the response better.
    
4.  Then we retrieve the relevant documents based on some logic.  
    This was pure semantic search before using vector embedding and cosine similarity, but has been changed since to use a hybrid system (Described later in this post)
    
5.  When asking the LLM to generate the response, we also fetch a system prompt that's in the codebase, which is sent along with our documents and instructions.
    
6.  If no relevant documents found then we just reply a message for that and end the turn, if not then we continue.
    
7.  If Streaming is enabled we call to the streaming response function, otherwise we generate the whole response at once and then return it.
    
8.  Also we remove source attribution from the LLM response itself because our pipeline explicitly marks it separately.
    
9.  After that, we update the conversation history so the latest turn is in the context for the next one.
    
10.  Finally we update the statistics for the whole turn so that we can track the metrics.
     
11.  Return the response and wait for the next call.
     

## Hybrid Search with BM25

If you've been following the previous posts, you know that the first thing i started with, was vector embedding and cosine similarity.

Related to vector spaces? Check. Using embedding models that are LLM based? Check!

So the next step would obviously be something even more sophisticated, even more enmeshed within the AI ecosystem right? Well, not quite.

To take several steps ahead, we need to take a few steps back, almost 40+ years ago.

We reach the age when the first personal computers were getting popular, but enterprise workloads were still the dominant use.

The internet was still not a thing yet, but people wanted to figure out how to search for keywords smartly across large volumes of data.

Artificial intelligence was in a limbo, one of the several AI winters as people talk about it, and it was the perfect time for statistical methods.

From this environment came out several Statistical keyword search algorithms, one of which, along with it's variants would go on to dominate search as Internet became a big deal.

Of course you build up your corpus of data by crawling and what not, but once you have it crawled, how do you decide which one to fetch? This algorithm comes in here.

This is where BM25, or as it's full name is, Okapi Best Match 25 comes in.

Now, below this you will see two crazy looking formulas that won't make much sense at first, but bear with me.

![](https://cdn.hashnode.com/uploads/covers/659a9af9ff6cf3c9cf4a9499/d6247ff4-9352-4b77-99bc-a378f916c639.png align="center")

![](https://cdn.hashnode.com/uploads/covers/659a9af9ff6cf3c9cf4a9499/7e4cb4d8-4b04-4f43-9fdd-b0ef0e99e575.png align="center")

You can of course look up tutorials on this online, including wikipedia which has a nice article on what each of these terms mean, but let me give you a crash course.

Essentially, this just codifies a way to calculate different things about a given query, `Q`. Such as,

1.  How frequently does the query word occur in the stored documents as a proportion of the documents? (The TF-IDF terms)  
    `The specificity of a term can be quantified as an inverse function of the number of documents in which it occurs.`
    
2.  How large or small those documents are? If we don't adjust for this, then we might bias for small documents where the query is perhaps one of the only words, while there can be larger ones which have more meaning, OR vice versa.
    

Now, moving beyond the formula, it is a lot more important that we can make it work alongside our vector search implementation, how do we do that?
